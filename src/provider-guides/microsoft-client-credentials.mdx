The Microsoft (Client Credentials) connector authenticates using OAuth2 client credentials with [admin-consented application permissions](https://learn.microsoft.com/en-us/entra/identity-platform/v2-admin-consent) instead of the standard per-user Authorization Code flow. A tenant admin approves your app once, and Ampersand can then access any user's Outlook mail, calendar, or contacts in that tenant without individual OAuth flows.

Please note that the standard [Microsoft connector](/provider-guides/microsoft) is a more convenient experience for individual consumers; it uses a standard OAuth 2.0 authentication scheme where each user logs in and approves the app. This connector is intended for builders who need bulk programmatic access across many users in a Microsoft 365 tenant.

## What's Supported

### Supported Actions

This connector supports:
- [Read Actions](/read-actions).
- [Write Actions](/write-actions).
- [Proxy Actions](/proxy-actions), using the base URL `https://graph.microsoft.com`.

### Supported modules & objects

The supported modules and objects are the same as the standard [Microsoft connector](/provider-guides/microsoft). See that guide for the full list.

```yaml
specVersion: 1.0.0
integrations:
  - name: microsoft-client-credentials-integration
    provider: microsoftClientCredentials
    ...
```

## Before You Get Started

This connector uses OAuth2 client credentials with admin-consented application permissions. There is no interactive login per user — your customer's Azure AD (Entra ID) tenant admin grants consent for your app once, and Ampersand authenticates as the app itself to access user data.

You will need:

- An Azure app registration with **application permissions** configured (e.g. `Mail.ReadWrite`, `Calendars.ReadWrite`, `User.Read.All`)
- The app's `client_id` and `client_secret`
- The customer's tenant ID (a GUID)
- Admin consent granted by the customer's tenant admin

If you already have a Microsoft OAuth integration set up on Ampersand, you can reuse the same Azure app — same `client_id` and `client_secret`. You just need the customer admin to grant admin consent for application permissions in addition to the delegated permissions they may have already approved.

<Note>This connector does not require a Provider App on Ampersand. The `client_id` and `client_secret` are provided directly when creating the connection, like other client-credentials providers (e.g. HelpScout).</Note>

### Configuring your Azure app for application permissions

If your Azure app was originally created for the standard Microsoft OAuth flow, you may need to add application permissions:

1. Go to [Azure portal](https://portal.azure.com) → **App registrations** → select your app.
2. Navigate to **Authentication**. Under **Supported account types**, ensure the app is set to allow multiple tenants, otherwise customer admins from other tenants will not be able to grant consent.
3. Navigate to **API permissions** → **Add a permission** → **Microsoft Graph** → **Application permissions**.
4. Add the permissions your integration needs (e.g. `Mail.ReadWrite`, `Calendars.ReadWrite`, `User.Read.All`).

### Getting admin consent

Provide your customer's tenant admin with the following URL:

```
https://login.microsoftonline.com/common/adminconsent
  ?client_id=<YOUR_AZURE_APP_CLIENT_ID>
  &redirect_uri=<YOUR_REDIRECT_URI>
  &state=<OPTIONAL_STATE>
```

When the admin visits this URL, Microsoft displays a consent screen listing all application permissions your app requests. After the admin approves, Microsoft redirects to your `redirect_uri` with `admin_consent=True&tenant=<TENANT_GUID>`.

The customer admin can also find their tenant ID in the Azure portal: **Entra ID** → **Overview** → **Tenant ID**.

<Note>After admin consent, it may take up to 5 minutes for permissions to propagate. If the connection returns authorization errors immediately after consent, wait a few minutes and retry.</Note>

### User targeting

Unlike delegated OAuth connections (which use `/me/...` endpoints), app-only tokens do not have a user identity. You must specify the target user in the URL path:

- `/v1.0/users/alice@contoso.com/messages` — Alice's messages
- `/v1.0/users/bob@contoso.com/calendar/events` — Bob's calendar events

This applies to both Proxy Actions and Read Actions.

## Using the connector

To start integrating with Microsoft (Client Credentials):
- Create a manifest file using `microsoftClientCredentials` as the provider.
- Deploy it using the [amp CLI](/cli/overview).
- If you are using Read Actions, create a [destination](/destinations).
- Have the customer admin grant admin consent (see above).
- Create a connection via the API with the tenant ID and your Azure app credentials:
  ```json
  {
    "provider": "microsoftClientCredentials",
    "providerWorkspaceRef": "<tenant_guid>",
    "oauth2ClientCredentials": {
      "clientId": "<azure_app_client_id>",
      "clientSecret": "<azure_app_client_secret>",
      "scopes": ["https://graph.microsoft.com/.default"]
    }
  }
  ```
- Start using the connector!
   - If your integration has [Read Actions](/read-actions), you'll start getting webhook messages.
   - If your integration has [Write Actions](/write-actions), you can start making API calls to our Write API.
   - If your integration has [Proxy Actions](/proxy-actions), you can start making Proxy API calls.
