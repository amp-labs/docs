---
title: "Microsoft (Client Credentials)"
---

This guide walks you through granting an integration access to your organization's Microsoft 365 data (Outlook mail, calendars, contacts) using admin consent. This is a one-time setup performed by a tenant administrator — individual users do not need to sign in or approve the integration.

## Before granting admin consent

You will need:
- **Azure AD (Entra ID) administrator access** to your organization's tenant. You must hold one of these roles: Global Administrator, Privileged Role Administrator, Cloud Application Administrator, or Application Administrator.
- The **admin consent URL** provided by the integration developer (it contains their app's client ID).

## Granting admin consent

### 1. Review and approve the consent URL

The integration developer will provide you with a URL like:

```
https://login.microsoftonline.com/common/adminconsent
  ?client_id=<APP_CLIENT_ID>
  &redirect_uri=<REDIRECT_URI>
```

1. Open this URL in a browser.
2. Sign in with your administrator account.
3. Microsoft displays a consent screen listing the permissions the app is requesting (e.g. "Read and write mail in all mailboxes", "Read and write calendars in all mailboxes").
4. Review the permissions carefully. Click **Accept** to grant consent.

After you accept, Microsoft provisions the app in your tenant with the approved permissions. The app can now access user data in your organization using its own credentials (no individual user sign-in needed).

<Note>If you see an error like "Need admin approval" or "AADSTS90094", verify that you are signed in with an account that has administrator privileges.</Note>

### 2. Provide your tenant ID

After granting consent, provide the integration developer with your **Tenant ID**. You can find it in:

1. Go to the [Azure portal](https://portal.azure.com).
2. Navigate to **Microsoft Entra ID** (formerly Azure Active Directory) → **Overview**.
3. Copy the **Tenant ID** (a GUID like `12345678-abcd-1234-abcd-123456789012`).

| Input | Where to find it |
|---|---|
| **Tenant ID** | Azure portal → Microsoft Entra ID → Overview → Tenant ID |

## After setup

Once admin consent is granted and the tenant ID is provided:
- The integration can access mailboxes, calendars, and contacts for any user in your organization.
- No individual users need to sign in or approve anything.
- The integration authenticates as itself (the app), not as any specific user.

## Revoking access

To revoke the integration's access to your organization:

1. Go to the [Azure portal](https://portal.azure.com).
2. Navigate to **Microsoft Entra ID** → **Enterprise applications**.
3. Find the integration's app in the list (search by name or client ID).
4. Click the app → **Properties** → **Delete**, or go to **Permissions** and remove specific permissions.

Alternatively, a Global Administrator can revoke consent via PowerShell:
```powershell
Remove-MgServicePrincipal -ServicePrincipalId <service-principal-object-id>
```

## Troubleshooting

### Authorization errors immediately after consent

Microsoft may take up to 5 minutes to propagate admin consent across all services. If the integration reports authorization errors right after you granted consent, wait a few minutes and try again.

### "Application not found in tenant" errors

This can happen if:
- The consent was not completed successfully. Try the consent URL again.
- You signed in with an account from a different tenant than intended.
- Your organization's Conditional Access policies are blocking the app.

### Access is broader than expected

Admin consent grants the app access to **all users** in your tenant for the approved permissions. There is no way to scope app permissions to specific users via the consent screen.

However, you can restrict app access to specific mailboxes using [Application Access Policies](https://learn.microsoft.com/en-us/graph/auth-limit-mailbox-access) in Exchange Online. This allows you to limit which mailboxes the app can access by creating a mail-enabled security group and associating a policy.

### Users without Exchange Online licenses

If a user does not have an Exchange Online license (e.g., they only have a Microsoft 365 F1 license without mailbox), the integration will receive a "MailboxNotEnabledForRESTAPI" error when trying to access that user's mail. This is expected — only users with active mailboxes can be accessed.
